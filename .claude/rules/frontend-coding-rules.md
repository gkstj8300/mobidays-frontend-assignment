## 2. 중복 작성 금지

- 새 함수/컴포넌트를 만들기 전에 기존 코드에서 동일 기능이 있는지 반드시 검색한다.
- 검색 없이 새로 작성하지 않는다.

---

## 3. 요청 범위만 작업

- 사용자가 요청한 변경만 수행한다.
- 주변 코드 리팩토링, 주석 추가, 네이밍 변경 등 요청하지 않은 "개선"을 임의로 하지 않는다.

---

## 4. API/라이브러리 사용

- 라이브러리 API를 사용할 때 해당 API가 실제로 존재하는지 확인한다.
- 확신이 없으면 사용자에게 버전/API 존재 여부를 확인한다.
- 추측으로 코드를 작성하지 않는다.

---

## 5. 인터페이스 변경

- 파일 간 인터페이스(함수 시그니처, 타입, props 등)를 변경할 때 해당 인터페이스를 사용하는 모든 파일을 확인하고 함께 수정한다.
- 확인 없이 한쪽만 수정하지 않는다.

---

## 6. 근본 원인 해결 (땜질 금지)

- 에러/실패 발생 시 근본 원인을 먼저 파악한다. 원인 파악 없이 증상만 제거하는 수정을 하지 않는다.
- 타입 에러를 `as any`, `@ts-ignore`, `!`(non-null assertion) 등으로 우회하지 않는다.
- 테스트 실패 시 테스트 코드를 삭제하거나 `.skip`으로 우회하지 않는다. 실패 원인을 수정한다.
- 린트/빌드 에러를 `eslint-disable`, 설정 완화 등으로 무시하지 않는다.
- 원인 파악이 어려우면 사용자에게 상황을 보고하고 판단을 요청한다. 임의로 우회하지 않는다.

---

## 7. Import 규칙

### import type 분리

- 타입만 가져올 때 반드시 `import type { ... }` 구문을 사용한다.

### 경로 alias

- 프로젝트 내부 모듈은 `@/` alias를 사용한다.

---

## 8. 데이터 통신 규칙

- `db.json`을 직접 import하지 않는다. 반드시 json-server 또는 msw 등을 통한 비동기 API 통신을 사용한다.
- API 예시: `GET /campaigns`, `GET /daily_stats`
- 원본 `db.json`은 수정하지 않는다.

---

## 9. 데이터 전처리 필수

- Division by Zero: CTR, CPC, ROAS 등 파생 지표 계산 시 분모가 0인 경우 안전 처리한다.
- Null Safety: `budget`, `endDate`, `conversionsValue`, `name` 등 null 가능 필드를 안전하게 처리한다.
- 데이터 정규화: 날짜 포맷(`YYYY-MM-DD` vs `YYYY/MM/DD`) 등 불일치 데이터를 정규화한다.

---

## 10. 작업단위 가이드라인

### 안전 작업단위

1회 지시에서 누락 없이 완료할 수 있는 안전 범위:

| 작업 유형 | 안전 범위 |
|-----------|-----------|
| 기능 구현 | 1기능 (1페이지 또는 1컴포넌트 단위) |
| 리팩토링 | 관련 파일 5개 이내 |
| 버그 수정 | 1이슈 |

### 위험 작업단위

아래 조건에 해당하면 반드시 분할하여 진행한다.

| 기준 | 예시 |
|------|------|
| 파일 10개 이상 동시 변경 | "모든 페이지에 로딩 스피너 추가해" |
| 범위가 열린 작업 | "전부 수정해", "모든 파일에 적용해" |
| 여러 feature 영역에 걸친 교차 변경 | "필터 + 차트 + 테이블 동시에 수정해" |

분할 방법: 변경 대상 파일 나열 → 기능 영역/파일 수 기준으로 단계 분리 → 각 단계를 별도 지시로 요청.

---

## 11. 지시 형식 권장사항

### 여러 이슈를 한 번에 처리할 때

번호로 명시적으로 나열한다: `3.1 글로벌 필터, 3.2 일별 추이 차트 수정해`

### 범위를 특정할 수 있는 구체적 지시어 사용

| 비권장 | 권장 |
|--------|------|
| "다 고쳐" | "캠페인 테이블의 정렬 기능 수정해" |
| "전체적으로 리팩토링해" | "글로벌 필터의 상태 관리를 Zustand로 분리해" |
| "모든 파일에 적용해" | "차트 컴포넌트 3개에 툴팁 추가해" |

### 대규모 작업의 단계별 분할

파일 10개 이상 또는 여러 영역에 걸치는 작업은 단계별로 분할한다.

## 12. 프로젝트 실전 교훈

### 교훈 요약

1. **원본 대조 필수**: 과제 명세서(`.claude/spec/MAIN.md`)를 반드시 먼저 읽는다.
2. **수정 전 실코드 확인 필수**: 문서/추론만으로 판단하지 말고 실제 코드를 읽어서 현 상태를 확인한 뒤 수정한다.
3. **라이브러리 export명 확인**: 아이콘·컴포넌트 이름을 기억에 의존하지 말고 패키지 검색으로 정확한 이름을 확인한다.
4. **데이터 예외 확인**: `db.json`에 의도적으로 포함된 예외 데이터(null, 포맷 불일치 등)를 반드시 전처리한다.
